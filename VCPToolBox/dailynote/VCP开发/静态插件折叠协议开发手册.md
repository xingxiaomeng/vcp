# VCP 静态插件与动态折叠协议开发手册

静态插件 (Static Plugins) 是 VCP 系统中提供基础全局状态感知的重要组件。与需要 AI 显式调用的由 `PluginManager` 拉起的“同步/异步指令型插件”不同，静态插件在 VCP Server 启动时作为**常驻子进程**被挂载。它们通过向标准输出 (`stdout`) 周期性或一次性输出特定格式的文本字符串、结构化 JSON，或共享折叠块文本协议，无缝地将外部状态（如实时天气、当前系统资源甚至本地音乐播放进度等）注入到所有 Agent 的运行时系统提示词（System Prompt）中。

在 VCP 中，静态插件的占位符（如 `{{VCPWeatherInfo}}`）在用户每一次对话发给 LLM 前，都会被 `messageProcessor.js` 的 `replaceOtherVariables` 核心引擎自动解析并替换。

本文档将详细阐述静态插件的基础构建方式，并重点介绍 VCP 独创的 **“动态上下文语义折叠协议” (Contextual Folding Protocol)**。

---

## 1. 静态插件基础概念

静态插件设计的初衷是**零心智负担注入**。只要插件进程（位于 `Plugin` 目录下并提供有效的 `plugin-manifest.json`）向 `stdout` 输出内容，主控 Server (`Plugin.js`) 就捕获并将其缓存在内存字典（如 `staticPlaceholderValues`）中。

**基础运行模式:**

1.  **启动**: Server 根据 `plugin-manifest.json` 中 `"pluginType": "static"` 的声明，跟随主服务一同被拉起执行。
2.  **输出**: 插件脚本完成外部 API 调用或系统指令查询后，简单地将结果 `print()` (Python) 或 `console.log()` (Node.js) 出口。
3.  **解析替换**: Agent 系统提示词如书写了 `{{VCP你的插件名}}`，则被自动替换为最新缓存的 `stdout` 字符串。

### 1.1 基础实现 (纯文本输出模式)

**适用场景**: 系统时间、轻量级资源占用显示、单句口号或永远只需要极短文本的场景。

**示例实现 (Node.js)**:
```javascript
// Plugin/TimeAnnouncer/TimeAnnouncer.js
function main() {
    const now = new Date();
    // 纯文本输出，直接被当做 {{VCPTimeAnnouncer}} 替换值
    console.log(`当前系统服务器精确时间：${now.toISOString()}`);
    process.exit(0);
}
main();
```

## 2. 动态上下文语义折叠协议 (`vcp_dynamic_fold`)

> **痛点**: 一些静态插件（比如 7天天气、完整个人日程表、每日长篇简报）返回的文本量巨大。如果用户当前的话题是“写一段代码”或“闲聊”，将数百 Tokens 毫无关联的详实日程表塞入 System Prompt，既严重浪费金钱（Token开销），也可能干扰 LLM 的注意力导致幻觉（Latent Space Collapse / Attention Hijacking）。

为了解决这一痛点，VCP 实现了深度整合 RAG 语义模块的“**自适应展示协议 (`vcp_dynamic_fold`)**”。

### 2.1 核心原理

1.  静态插件可以输出两类折叠协议：**结构化 JSON 协议**，或更轻量的**文本块协议**。两者最终都会被统一转换成 `vcp_dynamic_fold` 对象。
2.  VCP 会自动把 `plugin_description` 以及必要时的区块 `description` 发送给 Embedding 模型，进行高维向量化（Vectorization），并通过现有缓存/持久化链路复用。
3.  当用户与 Agent 交流时，VCP Server 会提取用户最后一次发出的 Query（例如“明天带伞吗？”）并实时计算其语句向量。
4.  系统根据折叠策略，对**用户 Query 向量**与**插件描述向量**或**区块描述向量**做余弦相似度匹配，并根据阈值，智能展开对应的 `content` 内容进行替换注入。

### 2.2 协议格式总览

当前静态插件有两种等价的折叠输出方式：

1. **JSON 协议**：适合程序内部已经是结构化对象的场景
2. **文本块协议**：适合手写说明文档、静态模板、或希望像 toolbox 一样维护折叠区块的场景

这两种协议最终都会通过共享解析模块 [`modules/foldProtocol.js`](modules/foldProtocol.js) 转成统一的 `vcp_dynamic_fold` 对象，供 [`modules/messageProcessor.js`](modules/messageProcessor.js) 的 [`resolveDynamicFoldProtocol()`](modules/messageProcessor.js:150) 处理。

### 2.3 JSON 协议（结构化输出）

当你的静态插件想要启用折叠功能时，可以向 `stdout` 打印合法的 JSON 字符串。

**示例实现 (WeatherReporter.js 概录):**

```javascript
const outputObj = {
  // 【必选】此标志位激活核心解析器的语义匹配引擎
  vcp_dynamic_fold: true,

  // 【必选】用于整体插件级语义匹配的自然语言描述
  plugin_description: "天气预报插件，提供各个城市及地区的实时天气、未来数天预报、空气质量预警与生活日出日落太阳位置",

  // 【可选但建议明确】折叠策略
  dynamic_fold_strategy: "toolbox_block_similarity",

  // 【必选】信息密度区块数组
  fold_blocks: [
    {
      threshold: 0.5,
      description: "未来多日天气、预警、空气质量和日出日落",
      content: "【实时空气】...【未来预警】...【未来24小时预报详尽版】...【未来7日预报详尽版】...【日出日落及月相数据】..."
    },
    {
      threshold: 0.35,
      description: "明后天天气和短期出行建议",
      content: "【实时概况】...【3日精简预报】..."
    },
    {
      threshold: 0.0,
      description: "默认天气概况",
      content: "【实时概况】空气质量: 良 (AQI 60)。(今日无预警)"
    }
  ]
};

// 输出给系统
console.log(JSON.stringify(outputObj, null, 2));
```

#### JSON 协议字段说明

- `vcp_dynamic_fold`：必选，固定为 `true`
- `plugin_description`：必选，整个插件的总描述
- `dynamic_fold_strategy`：可选，建议使用 `toolbox_block_similarity`
- `fold_blocks`：必选，区块数组
- `fold_blocks[].threshold`：必选，命中阈值
- `fold_blocks[].content`：必选，实际展开内容
- `fold_blocks[].description`：**强烈建议提供**，用于区块级精准召回；如果缺失，则退回到旧版“整体插件描述匹配”逻辑

### 2.4 文本块协议（推荐用于手写维护）

如果你不想手写完整 JSON，也可以直接输出文本块协议。这个协议现在已经被 [`Plugin.js`](Plugin.js) 原生兼容，和 toolbox 使用的是同一套解析规则。

**协议语法：**

```txt
[===vcp_fold: 阈值 ===]
区块内容

[===vcp_fold: 阈值 ::desc: 区块描述 ===]
区块内容
```

- `阈值`：必选，例如 `0.0`、`0.35`、`0.8`
- `::desc:`：可选，但**强烈建议填写**
- `区块内容`：直到下一个折叠头出现前的全部文本

**示例实现：**

```javascript
const output = `
[===vcp_fold: 0.0 ::desc: 默认天气概况===]
【实时概况】空气质量: 良 (AQI 60)。(今日无预警)

[===vcp_fold: 0.35 ::desc: 明后天天气和短期出行建议===]
【实时概况】...【3日精简预报】...

[===vcp_fold: 0.5 ::desc: 未来多日天气、预警、空气质量和日出日落===]
【实时空气】...【未来预警】...【未来24小时预报详尽版】...【未来7日预报详尽版】...【日出日落及月相数据】...
`;

console.log(output.trim());
```

#### 文本块协议的优势

- 不需要手工拼 JSON
- 适合把内容作为模板文本直接维护
- 与 [`modules/toolboxManager.js`](modules/toolboxManager.js) 使用同一协议，方便统一维护和后续升级
- `::desc:` 可直接驱动区块级语义召回

---

### 2.5 `description` 的实际作用

这是这次协议升级后最关键的一点。

#### 旧逻辑
如果区块没有 `description`，系统会主要使用 `plugin_description` 做整体匹配：
- 先判断这个插件整体是否和当前上下文相关
- 再按阈值决定哪些块展开

这种方式适合“整份内容都围绕同一主题”的静态插件。

#### 新逻辑
如果区块存在 `description`，系统会对**每一个区块描述单独向量化并匹配**：
- 用户问“改代码”“补丁”“写文件”时，只展开对应代码修改区块
- 用户问“浏览器页面验证”时，只展开对应浏览器自动化区块
- 用户问“查目录”“看文件”时，只展开对应文件读取区块

也就是说，`description` 不是注释，而是**精准控制召回的主开关**。

### 2.6 `vcp_dynamic_fold` 处理流源码细节（进阶参考）

如果你正在调试自己的插件无法正确折叠，可以按下面的实际处理链路理解：

1. **统一解析入口**：
   - [`Plugin.js`](Plugin.js) 会先尝试把静态插件输出解析为 JSON
   - 如果 JSON 中存在 `vcp_dynamic_fold`，则直接作为动态折叠对象保留
   - 如果不是 JSON，但检测到 `[===vcp_fold: ...===]` 文本块标记，也会自动走共享解析模块 [`modules/foldProtocol.js`](modules/foldProtocol.js)，转换成标准动态折叠对象

2. **共享协议模块**：
   - [`modules/foldProtocol.js`](modules/foldProtocol.js) 负责统一解析：
     - `parseFoldBlocks()`
     - `hasFoldMarkers()`
     - `buildDynamicFoldObject()`
   - [`modules/toolboxManager.js`](modules/toolboxManager.js) 与 [`Plugin.js`](Plugin.js) 现在共用这套逻辑

3. **向量缓存与召回**：
   - [`modules/messageProcessor.js`](modules/messageProcessor.js:246) 会对 `description` 或 `plugin_description` 取向量
   - 向量优先通过 RAG 侧缓存能力复用，避免重复 embedding 开销

4. **兜底回退 (Fallback)**：
   - 系统会自动寻找阈值最低的区块作为 fallback
   - 如果发生“数据库查不到”、“用户文本为空”、“RAGDiaryPlugin 不可用”等情况，系统不会崩溃，而会安全回退到最低阈值区块内容

## 3. Best Practices (最佳实践建议)

1. **粒度划分要合理**：建议静态信息的折叠内容至少维护三个层级。
   - `0.0`：极致浓缩。一行字能说明白的绝对不开两行。这是面对大多数无关场景时的静默底色。
   - `0.3`~`0.4`：实用摘要。包含近期最迫切需要被 LLM 摄取的核心内容。
   - `0.5`~`0.8`：全量展开。只在用户精准发问时释放。

2. **`plugin_description` 写整体主题**：它适合描述“这个插件总体是干什么的”。
   - 好例子：`展示当前播放的网易云音乐信息和进度`
   - 差例子：`Netease API stdout daemon wrapper`

3. **`description` 写区块具体意图**：这是精准召回的关键。
   - 好例子：`文件读取和目录浏览`
   - 好例子：`代码修改与补丁应用`
   - 好例子：`浏览器自动化与页面验证`
   - 差例子：`模块1`
   - 差例子：`高级内容`
   - 差例子：`其他说明`

4. **优先使用文本块协议维护长内容**：如果你的静态插件输出本质上是一大段说明文本、帮助文档、状态简报，优先考虑使用文本块协议，而不是在代码里手拼大 JSON。

5. **保留 `0.0` 兜底块**：无论 JSON 协议还是文本块协议，都建议至少保留一个最低阈值区块，避免在弱相关场景下完全失去基础感知。

6. **不要把 `description` 写成重复的 `plugin_description`**：如果每个区块的 `description` 都一样，就浪费了区块级语义筛选能力。应尽量让每个区块代表不同的能力面或信息主题。

7. **toolbox 与静态插件可共用一套写法**：如果你已经熟悉 [`modules/toolboxManager.js`](modules/toolboxManager.js) 对 toolbox 文档的折叠写法，现在可以直接把同样的 `[===vcp_fold: ... ::desc: ...===]` 语法用于静态插件输出。
